# AI クレジットシステム 設計

ユーザに毎月クレジットを付与し、AI 機能（現在見ている地域への質問、クイズ解答の補足など）を
使うとクレジットを消費する仕組みの設計をまとめる。無料会員よりプレミアム会員の方が毎月の付与
クレジット数が多い。

## 確定した設計判断

| 項目 | 決定 | 補足 |
|---|---|---|
| 提供面 | Web UI（地図画面）＋ MCP の両方 | **共通の残高プール**を両面で共有する |
| 推論の担い手 | Cloudflare Workers AI（サーバー側推論） | MCP 経由でもクライアントのモデルには委ねず、自前で推論・課金する |
| 消費粒度 | トークン従量 | 内部はトークン精度で計上する |
| 表示単位 | 抽象クレジット（丸めた数字） | 生トークン数はユーザに見せない |
| 残高不足時 | その月はブロック＋アップグレード誘導 | 追加購入・翌月繰越は現時点では行わない |
| 付与タイミング | 暦月一律（毎月 1 日） | 加入日基準の日割りはしない |
| 付与数 | 無料 < プレミアム | 具体値は「[要確定の数値](#要確定の数値)」参照 |
| MVP スコープ | 基盤先行 → AI 連携は後追い | 付与/消費/残高表示を先に固め、Workers AI 連携は次段階 |

> **なぜ「両面共有 × Workers AI」か**: MCP 経由でもサーバー側（Workers AI）で推論するため、
> Web UI・MCP のどちらから使っても同じ推論経路・同じ残高を通る。「MCP はクライアント（Claude 等）の
> モデルに委ねる」構成ではないので、MCP 利用時も実コストが自社に発生し、クレジット消費の対象になる。

## 既存資産との接続

新規に作る部分と、既にある仕組みに乗せる部分を区別する。

| 領域 | 既存資産 | 本機能での扱い |
|---|---|---|
| 会員区分 | `src/lib/billing/entitlements.ts` の `resolvePlan()`（`free` / `premium`） | 付与数の出し分けにそのまま使う |
| 課金基盤 | Stripe + better-auth（`subscription` テーブル、`plans.ts`） | 付与数を `plans.ts` に定数化。将来の追加購入は別 Price で拡張可能 |
| DB | D1（SQLite）+ Drizzle + `drizzle/` 連番マイグレーション | クレジット台帳・残高を同流儀で追加 |
| 月次バッチ | `daily_activity` の日次集計（運用系の前例） | 月次付与を Cron Trigger で追加 |
| MCP | `src/lib/mcp/tools.ts`（`list_aops` / `show_aop_map` 等） | 地域 Q&A ツール `ask_region` を追加（PR2 以降） |
| 推論 | **なし**（`openai` / `anthropic` / `workers-ai` いずれも未使用） | Workers AI をゼロから導入（`wrangler.jsonc` に `ai` バインディング追加） |

## データモデル

**台帳方式（append-only ledger）＋ 残高キャッシュ**を採用する。付与・消費・返却をすべて追記する
ことで、履歴表示・監査・二重消費防止を一枚岩で解く。

### `credit_ledger`（新規）

クレジットの増減をすべて記録する追記専用テーブル。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | text (PK) | `crypto.randomUUID()` |
| `user_id` | text (FK → `user.id`, cascade) | 対象ユーザ |
| `amount` | integer | 符号付き。付与は正、消費は負、返却は正 |
| `type` | text | `grant` / `consume` / `refund` |
| `request_id` | text (unique) | 冪等キー。同一リクエストの再送による二重計上を防ぐ |
| `period_month` | text | 付与・失効の対象月（例 `"2026-07"`） |
| `token_amount` | integer (nullable) | 内部精度の実測トークン（`consume`/`refund` 時） |
| `created_at` | integer (timestamp_ms) | |

> `amount` は表示クレジット、`token_amount` はトークン精度。表示クレジットは
> `token_amount` を[換算比](#要確定の数値)で丸めた値で、`amount` に確定させて保存する
> （表示のたびに再計算しない）。

### 残高キャッシュ（`credit_balance` 新規 もしくは `user` へ列追加）

台帳の SUM は毎回引くと重いため、現在残高をキャッシュする。台帳への追記と**同一トランザクション**で
更新し、台帳と常に整合させる。

| カラム | 型 | 説明 |
|---|---|---|
| `user_id` | text (PK, FK → `user.id`) | |
| `balance` | integer | 現在の残高クレジット |
| `period_month` | text | 残高が属する付与月。月が変わったら付与時にリセットされる |
| `updated_at` | integer (timestamp_ms) | |

> 独立テーブル（`credit_balance`）にするか `user` へ列追加するかは実装時に決める。
> クレジット関連の関心を 1 テーブルに閉じ込める観点では独立テーブルが素直。

### 付与数の定数（`src/lib/billing/plans.ts` に追記）

```
無料会員   : MONTHLY_CREDITS_FREE
プレミアム : MONTHLY_CREDITS_PREMIUM
換算比     : TOKENS_PER_CREDIT（内部トークン → 表示クレジット）
```

いずれも[要確定の数値](#要確定の数値)。定数化により、基盤はプレースホルダ値で着手でき、
後から値だけ差し替えられる。

## 消費フロー（予約 → 確定）

トークン従量では消費量が事前に確定しないため、**予約（reserve）→ 実測確定（settle）**方式を採る。
残高不足時はブロックなので、推論を開始する前に必ず残高を検証する。

1. **最低残高チェック** — 残高が最大見積を下回るならブロック。推論は開始せず、アップグレード誘導を返す。
2. **予約** — 最大見積分を `consume` として仮計上し、残高から引く（`request_id` 付き）。
3. **推論実行** — Workers AI を呼ぶ。
4. **確定・差分返却** — 実測トークンを換算し、予約との差分を `refund` で戻す。
   推論失敗時は予約全額を `refund` で戻す。

```
[残高チェック] → NG → ブロック（アップグレード誘導）
      │ OK
[予約: consume -max]
      │
[Workers AI 実行] → 失敗 → [refund +max]（全額返却）
      │ 成功
[settle: refund +(max - 実測)]（差分返却）
```

> **なぜ予約するのか**: 残高不足ブロックを「推論後」に判定すると、コストだけ発生して課金できない
> ケースが生まれる。先に最大見積を押さえることで、必ず残高の範囲内で推論する。

## 整合性・同時実行

Web UI と MCP の**両面から同一残高を同時に叩ける**うえに従量課金のため、二重消費・残高マイナスを
防ぐ設計が要になる。

- **冪等キー** — `credit_ledger.request_id` を unique にし、再送を弾く。
- **直列化** — ユーザ単位の **Durable Object** で残高更新を直列化するのが最も堅い。
  Workers の同時実行下でも、あるユーザの残高操作を 1 本に絞れる。
  - 代替案: D1 の条件付き `UPDATE`（`WHERE balance >= ?`）＋失敗時リトライ。
    追加バインディング不要だが、競合時のリトライ制御を自前で持つ必要がある。
  - **トレードオフは実装時に確定**する（DO 追加コスト vs リトライ実装コスト）。

## 月次付与（Cron Trigger）

暦月一律付与のため、**月 1 回の Cron Trigger**で全対象ユーザに当月分を付与する
（`wrangler.jsonc` に `triggers.crons` を新規追加。現状 Cron 設定は無い）。

- 付与は `grant` 台帳への追記＋残高キャッシュのリセット（`period_month` を当月に更新）を
  同一トランザクションで行う。
- `request_id` に `grant:{userId}:{period_month}` を用い、**当月分の二重付与を冪等に防ぐ**。
- **取りこぼし対策** — Cron 実行時に未ログイン等で対象外だったユーザは、初回アクセス時に
  「当月分が未付与なら遅延付与」するフォールバックを消費フロー入口に置く。

## UX

- **残高表示** — ヘッダ等に現在クレジットを表示。取得中のフラッシュ回避は `shouldShowAds` と
  同じ考え方（ローディング中は確定表示を出さない）。
- **消費履歴** — `credit_ledger` を元にした履歴ビュー（任意、後段でも可）。
- **残高不足導線** — ブロック時はプレミアムのメリット（付与数の多さ）を示すアップグレード誘導。
- **MCP 側のブロック表現** — クレジット不足を構造化エラー（`isError: true` ＋説明文）で返し、
  クライアントに「アップグレードで解決する」ことを伝える。

## 要確定の数値

以下は `src/lib/billing/plans.ts` の定数として持つため、後から値だけ差し替えられる。基盤 PR は
プレースホルダで着手可能。

1. **月次付与数（無料 / プレミアム）** — Workers AI の原価から逆算するのが正道。
   プレミアム（月額 300 円）の粗利と、想定利用回数から決める。
2. **トークン → クレジット換算比（`TOKENS_PER_CREDIT`）** — 表示クレジットの粒度。
   1 回の地域 Q&A がおよそ何クレジットに見えるかを、体感が自然になるよう調整する。
